## 1. Структура pages: `settings.vue` vs `settings/index.vue`
 
### Проблема
Документ утверждает (п. 16.3):
 
> `app/pages/settings/index.vue` — каркас с `UTabs` и боковым меню.
> `NuxtPage` в `settings/index.vue` для вложенных путей `/settings/profile`, `/settings/team` и т.д.
 
Фактически в проекте используется стандартный паттерн Nuxt «parent route + nested children»:
 
| Путь | Фактическое содержимое |
|---|---|
| `app/pages/settings.vue` | Каркас с боковым меню (`<aside>`), мобильным tab-row и `<NuxtPage />`. Именно здесь живёт layout, поиск по разделам, guard `role !== 'admin'`. |
| `app/pages/settings/index.vue` | Небольшой редирект (`navigateTo('/settings/general', { replace: true })`). Никакого `UTabs` и бокового меню здесь нет. |
 
### Почему важно
Любой читатель документа, пытающийся найти «каркас с табами», откроет `settings/index.vue` и увидит редирект. Это блокирует онбординг и ломает привязку требований к коду.
 
### Рекомендация
- В списке файлов п. 16.3 заменить:
  - `app/pages/settings/index.vue` — каркас с `UTabs` и боковым меню
  - → `app/pages/settings.vue` *(new)* — каркас nested-route: `<aside>` + мобильный tab-row + `<NuxtPage />`.
- Добавить отдельно: `app/pages/settings/index.vue` *(new)* — редирект `/settings → /settings/general`.
- Также синхронизировать ниже раздел «Новые/Изменённые файлы».

## 2. `UTabs` — заявлен, но не используется
 
### Проблема
В п. 16.3 и в разделе «Переиспользуемые утилиты» указан `UTabs` (Nuxt UI v3):
 
> На `< md`: `UTabs` горизонтально, контент ниже.
> **Nuxt UI v3:** `UCommandPalette`, `UModal`, `UTabs`, `USkeleton`, `UInput`, `UBreadcrumb`.
 
Фактически `settings.vue` реализован через набор `NuxtLink` c ручной стилизацией (строки 20–45 и 51–67). `UTabs` не импортируется и не используется ни в одной странице `settings/*`.
 
### Рекомендация
Либо:
- **(A) синхронизировать документ с реализацией**: заменить `UTabs` на описание «вертикальный nav из `NuxtLink` на ≥md; горизонтальный scrollable row из `NuxtLink` на <md». Убрать `UTabs` из «Переиспользуемых утилит».
- **(B) обосновать, почему `UTabs` заменён** (например, `UTabs` не даёт real-route-aware подсветку без костыля с `v-model` и ручным маппингом) и зафиксировать это решение в «Отклонениях».

## 3. Пункт меню «Интеграции» — перенос заявлен, но не завершён
 
### Проблема
Документ позиционирует integrations как таб внутри `/settings`:
 
> `app/pages/settings/integrations.vue` *(new)* — API-ключи, webhooks.
 
Фактически в репозитории одновременно существуют **два** маршрута, оба рабочих:
 
| Путь | Файл | Sidebar | i18n |
|---|---|---|---|
| `/integrations` | `app/pages/integrations/index.vue` (старый, 347 строк, старый UI) | Да, `Sidebar.vue:101` хардкод `label: 'Интеграции', route: '/integrations'` | Используется `nav.integrations` |
| `/settings/integrations` | `app/pages/settings/integrations.vue` (новый, по Epic 16) | Да, через `settings.vue:113` с ключом `settings.tabs.integrations` | Используется `settings.tabs.integrations` |
 
Итого: в сайдбаре остался **дубль** верхнеуровневого пункта «Интеграции», ведущий на старую страницу, а новая страница в настройках живёт параллельно. Это прямое противоречие замыслу «перенести в Настройки».
 
### Дополнительные наблюдения
- `app/components/app/Sidebar.vue:101` хардкодит русский label `'Интеграции'` (не через `t()`), в отличие от остальных пунктов.
- `app/components/app/MobileNav.vue` уже НЕ содержит пункта `/integrations` — то есть на мобильном перенос выполнен, а на десктопе нет. Это несогласованность UX.
- `i18n/locales/*.json` содержит как `nav.integrations`, так и `settings.tabs.integrations` — после завершения миграции первый ключ должен быть удалён.
### Рекомендация
Добавить в Epic 16 явные пункты удаления/зачистки и отразить их в блоках «Изменённые» и «Критерии приёмки»:
1. **Удалить** `app/pages/integrations/index.vue` (или заменить на редирект `/integrations → /settings/integrations`, если есть внешние ссылки).
2. **Удалить** строку `Sidebar.vue:101` (пункт `/integrations`).
3. **Удалить** i18n-ключ `nav.integrations` в `ru.json` и `en.json`.
4. **Критерий приёмки (новый):** «Пункт `Интеграции` не дублируется: в основной навигации его нет, он доступен только через `Настройки → Интеграции`».

## 4. Поиск внутри настроек (п. 16.4) — несоответствие реализации спецификации
 
### Проблема
Документ обещает полноценный поиск по ключам настроек:
 
> Индекс — массив `{ label: t('settings.theme.label'), tab: 'general', anchor: 'theme' }`.
> При клике по результату — `navigateTo('/settings/general#theme')` + scroll к anchor.
> Запрос `/` в поле фокусирует input (как в GitHub-settings).
 
Фактическая реализация в `settings.vue:118-122`:
```ts
const filteredNavItems = computed(() => {
  const q = settingsQuery.value.trim().toLowerCase()
  if (!q) return navItems.value
  return navItems.value.filter(item => item.label.toLowerCase().includes(q))
})
```
 
Это фильтр по именам **разделов** (General/Profile/Team/…), а не индекс по ключам конкретных опций. Anchor-навигации нет. Шортката `/` для фокуса нет.
 
### Рекомендация
Либо привести реализацию в соответствие с п. 16.4 (полный индекс опций + `#anchor` scroll + `/`-шорткат), либо переписать п. 16.4 как «фильтр разделов настроек» и явно вынести «поиск по ключам» в отдельный пункт будущего эпика.

## 8. Breadcrumbs в Header — хардкод вместо i18n (минорное, но смежное)
 
### Проблема
`app/components/app/Header.vue:94-107` содержит карту меток для breadcrumbs в виде русского хардкода:
```ts
const breadcrumbLabels: Record<string, string> = {
  dashboard: 'Дашборд',
  qr: 'QR-коды',
  ...
  integrations: 'Интеграции',
}
```
 
Это напрямую не ошибка документа, но Epic 16 в п. 16.3 декларирует «Breadcrumbs подтягивают текущий раздел», а локализации для `/settings/*`-путей через `t()` нет — в итоге при переключении языка хлебные крошки останутся русскими.
 
### Рекомендация
Добавить в документ явную подзадачу «локализовать `breadcrumbLabels` через `useI18n`» либо трактовать как отдельный follow-up и занести в `docs/splat-qr-ux-ui-review.md`.
